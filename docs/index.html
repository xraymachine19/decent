Built for the decentralized web, Decent makes using your own CSS-based style library easier than using Bigcorp's latest shovelware.

Setup and testing

  1. Run python -m SimpleHTTPServer in the decent directory.
  2. Follow the directions there (or just go to localhost:8000)
  3. Edit index.html or the less, then refresh to test stuff.
  4. index.html isn’t the public splash page for github but you can put that kinda content there if you’d like. It’s really just a scratchpad. Test-style setups (gray-scale contrasting table for example) are good here too.

  1. Run ‘lessc index.less css/index.css’.

Decent aims to preserve and emphasize accessibility.

How to write CSS with Decent

First, know Less

If you don’t know Less, the preprocessor for CSS, you should. Decent is written in Less.

Here’s a quick tutorial if you don’t know it.

Variables look like this:

p {
    @variable-name: #000000;
    color: @variable-name;
}

They’re set with the @ symbol. The above outputs:

p { color: #000000;}

Next, you can nest rules inside of each other:

p {
    @variable-name: #ffffff;
    span { 
        color: @variable-name;
    }
}

This lets you quickly write and organize rules in a codebase. The above outputs:

p span { color: #ffffff;}

There you go! That covers about 90% of the Less you will ever write. It has tons and tons great features besides these, though, and great docs to go with it. But if you know CSS and the above, then you know Less. Welcome to the club!

Wait, what about Sass?

We get it. A lot of people “moved on” to Sass.

We didn’t. And here’s why: Less is declarative and Sass is imperative.

Some people see Sass and think, “Great, a handy imperative layer in which to write and output code!”

We get that too. We’re just not those people. Generally, this group of people is looking to not write CSS. We even love that, in spirit. The best line of code is the line of code you don’t have to write.

But sometimes you do have to write and organize a whole lot of CSS into one codebase with variable outputs, different windows into your app’s UI world.

For this, we find Less’s declarative nature and lazy loading to be the optimal solution. We think of the imperative layer as constantly geting in our way. Less enhances and celebrates CSS. It doesn’t apologize for its declarative nature.

Still, if you know Sass, then you can use Decent pretty comfortably. The tutorial above covers most of the daily use difference. Just pretend you’re tweeting!

Seriously, though. No Sass?

No Sass. We’ve heard the features argument. It’s more about the paradigm for us.

If you want an example, check out this codepen. One of the two examples will make sense to you. If the boxes are green, you must go lean! (…er stylesheets)

If you’re constantly fighting to namespace components and cart around CSS in Javascript, you might be downstream of this problem. Decent lets you build a modern UI style codebase with centralized style controls and easy support for building multiple app/experience/portal targets.

A single Decent codebase can put out consistent looks/stylesheets for your web app, your Electron app, your marketing page and your Myspace widget (or more recent technology). All without running into those weird global scope issues as the codebase gets more complex. All without walking on eggshells around that global scope after being tortured by Kafkaesque errors.

How do I get started?

First, you should have some sort of web page to look at. It can be an index.html file on your desktop. Or a Django app running in the cloud. We really like Sitepoint’s starting point because we’re copy/paste bandits: https://www.sitepoint.com/a-basic-html5-template/

Take note of where it links to the CSS: styles.css is your stylesheet target.

Don’t write anything in there. You don’t edit CSS files now. You are writing in the Decent way now.

Replace the <body> element with this (no, we won’t need that scripts file— Decent is not a Javascript library):

<body>
    <main>
        <section>
        </section>
    </main>
</body>

Out of the box, Decent uses main, section and nav as high-level organizational elements. We’ll write our demo HTML in that section element.

Second, you need a way to look at this webpage. If you need something quick, we recommend:

python -m SimpleHTTPServer

That’ll serve your index.html file locally, acting like a real website, provided you run it in the same folder (the desktop folder, in the example above). You might need to install python.

Next, we’ll need NPM and Less. Decent is the thing that runs your CSS codebase. Decent runs on Less which runs in NPM. It’s less complicated than it sounds. Just run:

npm install -g less

That installs Less globally, meaning you can run it in any directory. You want to do that because Less runs on your machine, and not so much as a featured package on your site. Your site only shows the compiled results of Less. (The exception here might be less.js. We don’t recommend using less.js with Decent.)

If you don’t have NPM, get it here: https://www.npmjs.com/get-npm

It’s worth it! Even if you only ever use it for Less.

A simple workflow

If you’re following along with the above, your directory structure might look like this:

project-name
├── index.html
└── styles.css

If you’re using a more complicated setup, you can probably extrapolate from this demo. Less integrates well with command-line based workflows and has desktop solutions too.

Changes to both files will be reflected on http://localhost:8000/ whenever we make them and refresh. So where do we put Decent?

The answer: Anywhere you’d like! It works best right in the root directory of the site.

project-name
├── decent
│ ├── index.less
│ └── (many, many other files)
├── index.html
└── styles.css

At this point, you may want to change the name of the ‘decent’ folder into something more sensible, like ‘less’:

project-name
├── less
│ ├── index.less
│ └── (many, many other files)
├── index.html
└── styles.css

In this way, Decent is your CSS/Less codebase. Or at least the start of your codebase.

Wait, can’t I just grab the node package and import?

This section is a little advanced. To continue with the directory building in the simple tutorial, click here.

You can! We recommend using Decent in the above way, as a downloaded and bootstrapped CSS codebase starting point.

But you can also grab the Node package and import it in your Less codebase. Decent files are all designed to be able to be imported on their own, with their own dependencies stated inside their file.

Want Decent’s awesome contrasting superpowers but nothing else? Try:

@import "node/directory/decent/mixins/contrast.less";

Importing the entire library can be a little risky. Decent is meant as an environment, and may have problems sharing the stage in this capacity.

When it came to easy variable names or the sanctity of the scope, we picked the easy variable names every single time.

The "golden path" install is installing with node, then copying the folder over to serve as your codebase. That way you have a local copy of the original files, along with access to Decent's node dependencies and component imports. (You may need to change the @node_dir variable)

Let’s run this thing!

Where were we? We had just put together our basic directory structure:

project-name
├── less
│ ├── index.less
│ └── (many, many other files)
├── index.html
└── styles.css

Where do we start with the actual styling? Let’s imagine we want to write this rule:

p { border: 2px dashed red;}

In CSS, we start writing the rule. In Less, we start writing the rule.

In Decent, we add the rule to the p definition already in elements/p.less:

project-name
├── less
│ ├── index.less
│ ├── (many, many other files)
│ └── elements
│   ├── p.less
│   └── (other elements)
├── index.html
└── styles.css

It's quicker and easier! If you're in Sublime, Atom or VS Code, you can use ctrl/cmd + P and type "p.less" to get to the file quickly.

This happens because Decent makes it so that your website supports all HTML. All HTML elements have their own file and represenation in Decent. HTML support isn’t a weird feature for Decent to have; it’s a weird feature for most sites to be lacking.

Workflow-wise, this means finding “p.less” through your IDE and editing the lone rule within the file, instead of scrolling through one big file looking for one rule among dozens.

For everything outside of HTML elements, we can add rules in the traditional way. For the demo purposes, we can write new rules at the end of index.less. Writing rules here gives us easy control over the variable environment. Lazy loading uses the definitions it finds last, so writing Less here is kinda like having “the high ground.”

We might write this rule at the end of index.less, after all the import statements, instead of the paragraph rule:

.offer-frame { border: 2px dashed red;}

Now our .offer-frame component (or whatever it is) has the border rule. How do we get it to show up? First let’s make sure it’s in our index.html in the section inside body:

<p class="offer-frame">
    20% off compile times!
</p>

It won’t show up if it’s not there, right? Then let’s compile the rules for how it looks using Less.

Compiling the Less

To compile our Less, we need to be in our terminal in the site’s root directory, which here is “project-name.” Look, all the best frameworks are boring. Bear with us.

project-name
├── less
│ ├── index.less
│ └── (many, many other files)
├── index.html
└── styles.css

To compile here, the command is:

lessc less/index.less styles.css

No, that’s not a typo. It’s lessc in the command line. This will compile the CSS and report back on whether it was a success or a failure. If it’s a failure, we’ll get a hint as to why. If it’s a success, we’re ready to look at the changes.

Going back to http://localhost:8000/, we should see our new dashed red border line on the offer component.

Now that it’s there and working, let’s make it official. To do that, we might add a ‘components’ directory as a sibling of the ‘elements’ directory, with an ‘offer-frame.less’ file in there. Then we’d import that in index.less instead of just writing the rules right there.

Or we could just leave it where it is. It’s just a demo. In general though, the above pattern is recommended, specfically a ‘components’ directory and a ‘pages’ directory. We urge you to do this now, just as an exercise, to get comfortable editing your Less codebase. Which is what Decent is as soon as you copy it over.

Since a barebones approach is better for many projects, and since the tech world has so so many names for “components” and “pages,” we decided to leave that bit of directorial prescriptiveness out.

Writing Less the Decent way

Let’s edit .offer-frame.

Let’s give it a red background:

.offer-frame { 
    border: 2px dashed red;
    background: red;
}

Compile with lessc and see the red background. There are two problems here, both of which we will solve the Decent way.

First, let’s get the text contrast up to standards.

We could write out something like color: white; or we could find something that worked well with the design. Or we could just automate all of that as a part of our project settings (This is what Decent does).

How do we know what color some text is, anyway? If the designer looked up the contrast numbers, then the designer wasted his time. If the developer looked up the contrast numbers, then the developer wasted her time. Instead, use Decent to establish a contrast standard along with a tinting/shading strategy unique to your project.

To do this, we use the .contrast() mixin to programmatically find the best text color given the CSS property and background color as an argument. It takes no time at all (except, well, compile time) and provides perfect consistency (which we can adjust later!) across a project.

So here we can write:

.offer-frame { 
    border: 2px dashed red;
    background: red;
    .contrast(red);
}

This will output a color that meets our site’s default contrast standard. Speaking of which, Decent has a whole bunch of defaults. One of them will be good for the problem we see on refresh: The .offer-frame paragraph has no padding.

Naturally, we add this in:

.offer-frame { 
    border: 2px dashed red;
    background: red;
    .contrast(red);
    padding: 
}

But what comes next? What do we set the padding to? We set it to the default padding, which is set at the project level. But how do we express that? In CSS, it’s maybe not so straightforward. In Decent, it looks like this:

.offer-frame { 
    border: 2px dashed red;
    background: red;
    .contrast(red);
    .default(padding);
}

Ultimately, this ‘padding’ reference goes back to Decent’s properties.less file which contains default variable counterparts for every single CSS property, meaning .default() can be used with any CSS property. There are no eggshells here. Decent unapologetically puts a lot of variables into the global scope. .default(border), .default(margin), .default(box-shadow) will all output rules assigning those CSS properties to those same-named Less variable values.

When transplanted to another app, .offer-frame will play by that app's rules, using that app’s default padding. This idea of transplantability is a core theme in Decent, which targets the decentralized web. A shared precompile styling environment lets the sites of the next Internet share parts as needed without looking ridiculous. (Don’t worry, it works with the "Legacy Internet" too.)

Enough of the ideals. Let’s clean up .offer-frame a bit.

Color and meaning

Generally @color is used to represent what color something is.

It’s not, as implied above, the default value for “color:” the CSS property. It’s the only exception, because we shouldn’t be writing “color:” in our code.

So let’s set that red background to a variable:

.offer-frame { 
    @color: red;
    border: 2px dashed @color;
    background: @color;
    .contrast(@color);
    .default(padding);
}

Now it has padding, the text is properly contrasted, and it even has that neat border.

The problem now is that, in most themes and applications, red (the CSS color) is just a bit too much.

There’s a less obvious problem: Red means something to us. But it doesn’t necessarily mean something in general.

Project-wide meaning is a hard thing to nail down, especially in a visual design sense. To remedy this, Decent ships with ten semantic colors, all of which convey some sort of meaning. Here, we’ll convey meaning through the variable @failure, which is always red.

.offer-frame { 
    @color: @failure;
    border: 2px dashed @color;
    background: @color;
    .contrast(@color);
    .default(padding);
}

Notice that the contrast adjusts to the new color. If we changed it to @success, which is always green, the text contrast would still just work. Try ‘white’ and ‘black’ too.

Note also that the semantic colors in your codebase are all downstream of the @base-color variable. This variable should be something like your “brand color” or main color of the project. There are as many sets of semantic colors as there are input values.

That’s it! You’re done.

Your .offer-frame may not seem impressive on its own, but let’s review what you’ve done here:

  • You learned (or brushed up on) some Less basics
  • You bootstrapped a fully-featured CSS codebase
  • You gave something semantic meaning through color
  • You implemented and wieled project-wide defaults
  • You learned how to not worry about text contrast

Okay, maybe it is another component library. A little bit. These are just to get you started on your own set.

Decent works great with any Javascript project... because it doesn't include any Javascript of its own, and therefore can't conflict with whatever JS approach your project uses.

Decent's components are written to be simple and composable, meaning they can usually be used with the other component classes, or with CSS property-name classes.
